# 从“模型看懂”到“镜头可交付”：ShotSeek 的十日谈

## 一句话定位

ShotSeek 是运行在 NVIDIA DGX Spark 上的证据对齐长视频场景检索 Agent。用户描述
一句对白、一个动作或一个剧情关系，系统返回原片中的准确镜头边界，并给出画面、
对白和 Agent 决策证据，最后导出为剪辑流程可以继续使用的数据。

我们从一开始就没有把它定义成视频聊天。聊天回答“发生了什么”就可以结束，
影视后期却需要回答三个更严格的问题：

1. 发生在完整节目时间线的哪一帧？
2. 为什么相信这次命中？
3. 结果能否进入下一步剪辑？

这三个问题决定了 ShotSeek 的技术路线。

## 第一天：先验证真实接口，不先画产品

项目的第一个里程碑只有一个目标：证明 StepFun 的视频、文件和音频接口能在当前
环境中稳定工作。我们没有先做前端，也没有先拆很多 Agent，而是冻结输入视频、
请求参数、原始响应、Schema 和时间换算。

这个顺序很朴素，却避免了一类常见风险：页面已经很完整，核心模型调用仍依赖
未经验证的假设。真实接口通过后，脱敏 Fixture 只用于回归和离线演示，并明确标记
为 `CACHED`，不冒充 `LIVE`。

## 第二天：建立统一时间基

模型返回的时间不是剪辑时间。视频切片有自己的局部时间，音频识别有毫秒时间戳，
原片还有帧率、镜头切点和容器时长。ShotSeek 因此同时保存：

```json
{
  "start_ms": 812340,
  "end_ms": 829880,
  "start_frame": 19496,
  "end_frame": 19917
}
```

所有模型事件先从切片局部时间换算到原片全局时间，再映射到真实 Shot Grid。
展示层才把整数时间格式化为时码。毫秒用于 API 和播放器，帧用于边界审计和
CMX3600 EDL。

## 第三天：模型语义不能直接成为数据库事实

Step 3.7 Flash 负责把视频片段解析成结构化视觉事件，StepAudio 2.5 ASR 负责
对白、分句和时间戳。但模型输出只是候选事实。

每个响应都要经过 Schema 校验、时间越界检查、重叠去重、低置信标记和原始响应
留档。最终 Scene 必须能反向追溯到模型片段、对白或视觉证据和真实镜头边界。

这形成了 ShotSeek 最重要的资产：Evidence Timeline。换模型、换查询规划器或换
索引时，Scene 和证据契约仍然稳定。

## 第四天：Search Core 不是一个向量相似度

精确对白、视觉动作、“第一次”和“之前”不是同一种检索问题。ShotSeek 先把查询
解析成 QuerySpec，再执行 SQLite FTS5 宽召回、确定性时间运算和证据验证。

“第一次发现某物”不能先取普通 Top 3 再挑最早。正确流程是先召回全部有效候选，
完成角色和证据过滤，按原片时间排序，最后执行序数约束。模型可以理解语言，
但不能猜时间关系。

## 第五天：Agent 要留下可审计轨迹

我们没有让多个 Agent 在界面上互相“讨论”，而是把系统实现成有类型约束的数据流：

```text
Query Planner
→ Top 20 Retriever
→ Temporal / Ordinal Resolver
→ Evidence Verifier
→ Shot Boundary
→ Final Result
```

每个节点都记录输入、输出、候选数量、过滤原因和分项得分。评委看到的不只是一张
结果卡，而是系统为什么选择这个镜头、为什么排除其他候选。

## 第六天：从脚本进入 Production Runtime

算法闭环后，最大的变化不是再加一个模型，而是把 `python run_x.py` 变成：

```text
Upload → Job → Progress → Result
```

Production Runtime 提供内容寻址上传、持久化状态机、SSE 进度、取消、阶段重试和
重启恢复。一个媒体 Worker 负责串行媒体阶段，视觉 API 在受控并发内运行。浏览器
通过 HTTP Range 播放代理视频，可以直接跳转长片而不需要完整下载。

## 第七天：把证据放进产品，而不是日志

工作台首屏只保留播放器、搜索、最佳结果和必要状态。Visual、Dialogue、Boundary
和 Agent Trace 收进 Evidence Drawer，需要时再展开。这样既保留技术深度，也不让
GPU 指标和原始 JSON 淹没用户任务。

结果可以导出 JSON、SRT、XML 和 CMX3600 EDL。导出模块与 UI 使用同一 Scene 和
帧级时间基，避免“页面显示一个时间，交付文件又是另一个时间”。

## 第八天：真实长视频让问题暴露出来

项目完成了 36:58 连续素材的真实 Production Runtime：

- Step 3.7 视觉产物为 `LIVE`；
- StepAudio 2.5 ASR 产物为 `LIVE`；
- 生成 216 个 Scene；
- 总用时 1784.916 秒，RTF 约 0.805；
- Registry SQLite 完整性检查通过。

另一个 02:26 独立素材在 144.770 秒内生成 18 个 Scene，RTF 约 0.992。

第一次运行还暴露了一个媒体契约问题：视频恰好是 3650 帧、25fps，也就是
146.000 秒；AAC padding 让 MP4 容器时长变成 146.048 秒。旧探测器把容器时长
和视频帧数比较，误判 CFR。修复后，系统用视频流时长和帧数做校验，仍保留对真实
不一致的拒绝，并增加回归测试。

这类问题很难在模型 Demo 中出现，却是产品运行时必须解决的细节。

## 第九天：黄金样片全对，不等于系统泛化

黄金回归 40 条查询的 R@1 和 R@3 都是 100%。这证明固定链路没有倒退，但不能证明
系统可以泛化到任意影视素材。

我们在首次运行前冻结 Development、Holdout 和 Longform 数据集。结果是：

| Split | R@1 | R@3 | 负例误报 |
| --- | ---: | ---: | ---: |
| 黄金回归 | 100.0% | 100.0% | 0 |
| Development v1 | 100.0% | 100.0% | 0 |
| Holdout v1 | 66.7% | 66.7% | 0 |
| Longform v1 | 73.3% | 73.3% | 0 |
| Holdout v2 首次运行 | 16.7% | 16.7% | 0 |

总体状态因此是 `MIXED - GENERALIZATION GATES NOT MET`。

Holdout v2 只返回两个结果，两个都正确。系统没有乱答，却漏掉了 llama、animal、
goat-like animal 的指代变化、复合动作和序数表达。这说明 Rule Verifier 的边界是
precision-first。我们没有修改冻结查询、补接受 Scene，或者把失败集重新命名为
通过。

## 第十天：比赛作品最重要的是可信

ShotSeek 当前可以真实展示：

- 上传视频并运行完整 Job；
- DGX Spark 本地媒体、镜头、时间线、索引和 Agent Runtime；
- StepFun 视觉与 ASR；
- 自然语言搜索和播放器精确跳转；
- 画面、对白、边界和 Agent Trace；
- JSON、SRT、XML 与 EDL 交付；
- 36:58 规模运行和可复现评测。

它不能宣称所有独立泛化门禁已通过。下一步可以把 Holdout v2 变成 Development v2，
设计跨实体别名和模型 Top-K 复核，再用完全未见的 Holdout v3 验证；但首次失败结果
会永久保留。

项目最后形成的共同理解是：

> StepFun 负责理解“视频里发生了什么”，DGX Spark 和 ShotSeek 负责确定“准确发生
> 在哪个镜头、哪一帧”。

ShotSeek 不是让模型说得更像答案，而是让每个答案都能被证据、时间线和交付文件
共同约束。对影视后期来说，这才是从模型能力走向生产工具的关键一步。
