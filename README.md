# ShotSeek

> 一句话，定位长视频里的任意场景。

ShotSeek 是运行在 NVIDIA DGX Spark 上、由 StepFun 多模态模型驱动的影视长视频镜头定位 Agent。它面向剪辑师和影视后期团队，把长视频转换为可搜索、可验证、可交付的结构化场景时间线。

## 为什么做 ShotSeek

长视频后期制作中，剪辑师经常需要寻找某句对白、某个人物动作或某个剧情节点。传统方式依赖拖动时间线、反复播放和人工记忆，素材越长，查找成本越高。

ShotSeek 的唯一核心价值是：

> 把自然语言描述转换为可直接定位、复核和交付的场景时间码。

例如：

> 找到女主第一次发现尸体的场景。

系统应返回最佳命中与少量候选结果，并提供：

- 入点和出点时间码
- 关键帧与片段预览
- 一句话匹配理由
- 视频理解、对白、画面文字或剧本等证据
- 可继续用于后期流程的结构化导出

## 产品定位

ShotSeek 是面向影视后期的证据驱动场景定位 Agent，不是通用视频聊天工具，也不是监控视频分析系统。

它强调三点：

1. **场景可定位**：结果必须落到完整节目时间线中的明确位置。
2. **结论可验证**：每次命中都带画面、对白或文本证据。
3. **结果可交付**：搜索结果能够导出为标准化数据，而不只停留在聊天回答。

## 技术分工

### StepFun

第一版核心 AI 能力全面通过 StepFun API 接入：

- **Step 3.7 Flash**：视频理解、关键帧文字提取、中文意图理解、检索规划、结果解释与 Agent 编排。
- **StepAudio 2.5 ASR**：语音识别、分句、字词级时间戳与说话人信息。
- **StepFun Files/API**：上传并复用视频切片、音频和关键帧。

原始视频切片、音频和关键帧会根据任务需要上传至 StepFun。画面文字由多模态视觉理解提取，不宣传为独立专用 OCR 服务。

### NVIDIA DGX Spark

DGX Spark 是本地高性能视频工作站，计划承担：

- GPU 视频解码、转码、切片、抽帧和预览生成
- 长视频时间码维护与多片段结果合并
- StepFun API 并发调度、重试、缓存和任务状态管理
- 场景索引与低延迟检索
- Web 服务、交互界面和交付文件生成
- GPU 利用率、处理耗时和流水线状态采集

具体采用的 NVIDIA 编解码与 GPU 检索组件，将根据本机环境实测后确定。

### OpenBMB

现有 OpenBMB 项目作为主要工程基础，计划吸收其中已经验证且有权开源的能力：

- 长视频任务管线
- 视频切分与镜头处理
- 时间码归一化
- 场景结构化
- 缓存和任务状态
- JSON、XML、SRT 导出
- 可复用的服务端结构

原项目保持只读，参赛实现只在本仓库进行。

### NVIDIA VSS

NVIDIA VSS 只用于吸收 Agent、任务状态、视频时间线和 UI 组织思路，不复制其品牌、素材、产品界面或完整实现。

## 第一版输入

- MP4 视频
- MOV 视频
- 可选 TXT 剧本
- 可选 DOCX 剧本
- 可选 SRT 字幕

长视频会在 DGX Spark 上转码并切分。单个 StepFun 视频片段按小于 128MB 的 MP4 设计，同时保留原始节目时间码偏移。

第一版不承诺直播流、网盘链接、任意编码格式或多集批处理。

## 第一版输出

页面结果至少包含：

- 场景描述
- 入点与出点时间码
- 关键帧
- 片段预览
- 匹配理由
- 证据来源
- 置信度或排序分数
- 播放器跳转动作

计划支持的文件导出：

- JSON
- XML
- SRT

EDL 是核心链路稳定后的加分项，不属于第一版硬性承诺。

## 界面原则

界面采用深色、专业的影视后期工作台风格，但不复制 DaVinci Resolve 或 NVIDIA VSS。

首屏只突出：

1. 视频
2. 搜索
3. 最佳结果

详细的 ASR、画面文字、剧本对齐、置信度和请求记录通过证据抽屉按需展开。API 耗时、GPU 状态和任务日志进入诊断页，不干扰剪辑师的主要操作。

“商业化界面”在本项目中指成熟、克制、详略得当的信息层级，不是加入收费或经营面板。

## 主演示流程

上传素材 → 查看分析进度 → 输入自然语言 → 跳转目标场景 → 展开匹配证据 → 导出剪辑数据

演示由两部分组成：

- **实时闭环**：约 60 秒素材真实调用 StepFun API。
- **规模证明**：使用预先处理的约 43 分钟完整剧集展示长视频能力。

如果现场 API 失败，界面必须明确标记失败并切换缓存演示，不得把缓存结果伪装成实时生成。

## 验收目标

以下均为目标值，只有经过实测后才能作为项目成果对外报告。

| 指标 | 目标 |
| --- | --- |
| 人工标注查询 | 至少 20 个 |
| 场景检索召回 | Recall@3 不低于 80% |
| 命中标准 | 返回时间点位于人工标注的目标场景区间 |
| 查询延迟 | 索引完成后 P95 小于 3 秒 |
| 现场处理 | 60 秒素材从上传到可搜索不超过 5 分钟 |
| 导出一致性 | JSON、XML、SRT 的场景数量和时间码一致 |

## 第一版不做什么

- 不自动生成最终成片
- 不直接控制 DaVinci Resolve
- 不做人脸身份识别或演员实名识别
- 不处理直播流
- 不做多集批处理
- 不训练或微调模型
- 不做通用监控视频分析
- 不做视频生成、自动配音或数字人

## 数据与安全

- 视频切片、音频和关键帧可能上传至 StepFun API，产品必须在上传前明确提示。
- API Key 只能通过本地环境变量或安全配置注入。
- 密钥不得进入前端代码、日志或 Git 仓库。
- 日志中的请求信息必须脱敏。
- 示例素材必须具备比赛演示与公开展示权限。
- 运行缓存、任务数据和生成物只允许写入本项目目录树。

建议的本地密钥变量名称：

~~~text
STEPFUN_API_KEY=your_key_here
~~~

本地环境文件已由 .gitignore 排除。后续可提供不含真实密钥的 .env.example。
